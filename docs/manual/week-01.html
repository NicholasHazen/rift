Weeks · Core loop

The humble resource node

Week 01Phase: Core loopBudget: 5 sessionsresolution: full
Chronicle fragment, run [unlogged] — First vein depleted; one chunk carried to the crate at a waddle. Total haul: one rock. The award for Heaviest Load went uncontested.

By Friday you will have a resource node that two networked clients can harvest, break, and haul, and you will have felt the dungeon-master model — the server as the game's only storyteller — actually work instead of reading about it. Everything in the rift is downstream of one primitive, a thing in the world players extract value from: ire (the currency greed mints), the haul, the greed dial, and the exit decision itself all flow out of it. This week, that one primitive forces the entire networking core through your hands.

▣ Gate — definition of done

The test topology is two PIE clients (Play In Editor — the game running inside the editor) plus the dedicated-server checkbox, so the world is owned by a headless server nobody plays on. Both clients see the same node break, the same chunk pop, and a carried chunk deposit correctly, and the whole chain is server-authoritative end to end — including a client that grabs a chunk another client dropped. Carrying visibly slows you, the cheater test (this week's exam, defined below) fails safe, harvesting already feels good, and Clip #002 is posted.

The week opens with one prerequisite listen and then five build sessions. Before S1, hear the dungeon-master page through once, start to finish. It is a listen, not a session — the commute, the dishes, the stretch while week 0's engine downloaded — and it is the longest listen in the manual. This week is its lab, so every session below links back into it by anchor for the partial re-listen.

S1 and S2 are yours alone. S1 holds the codebase's first replicated property, a variable the server copies down to every client, and S2 holds its first RPC (a remote procedure call — a request one machine sends another). Both are first-of-pattern, and first-of-pattern belongs to your hands (rule 4). Every replicated property and every RPC ships with a one-line authority explanation the day it's written, and the habit starts with the first one. The trap this week: substituting input for output. You'll know you're in it when a second tutorial starts before AResourceNode compiles. The move: build S1 first and read with a bug in hand, because the shelf pulls below — the week's three curated links — are for questions, not preparation.

The sessions

Five sessions build a single actor and everything that follows from it: a fact, a request, a break, a carry, and a proof with juice on top. Each item carries its own done-when. All week, the fleet's standing job is to log the first entries into UE-GOTCHAS.md (the studio's running record of hard-won fixes) as they're earned, and this week reliably earns some.

S1 — The fact

[NICK — by hand] Hand-write AResourceNode: a Health property declared UPROPERTY(ReplicatedUsing=OnRep_Health), registered in GetLifetimeReplicatedProps, every mutation guarded by HasAuthority(). Give the server a debug key that ticks health down, and watch both client windows react through OnRep_Health, the client-side hook that runs when the replicated value changes. The point of the session is the feeling: you pressed a key on the server, and the world told the clients. Why can't a client just decrement the health itself? → concept-netcode.html#authority. Done when: a server-side health change renders on both clients, and the property carries its one-line authority explanation.

S2 — The request

[NICK — by hand] This is the project's first client→server request, and both of the week's silent failures live in this session, so build it with the links open. The first failure is about where the RPCs live. Server_StartHarvest and Server_StopHarvest belong on your Character or PlayerController — anywhere on the spine the client owns — and they take the node as a parameter rather than sitting on the node itself. That placement matters, because a Server RPC called on an actor the client doesn't own is dropped with no error, no warning, and no log line. Where may a Server RPC live, and why does the wrong answer silently do nothing? → concept-netcode.html#ownership. The second failure is about the RPC's shape. Send the pair Reliable so the engine guarantees each call arrives, fire one on key press and one on key release, and let the server tick the drain in the interval between them. What you must never do is send an RPC on every held frame, because a reliable call every frame is a buffer overflow on a timer, and the bill is paid out as mystery disconnects. Why events, never streams? → concept-netcode.html#reliability. Write the validation — node alive, player in range — as if the client is lying to you, because someday one will be. Hold-to-channel is the harvest verb for now, and whacking proper arrives in week 2, once melee exists. Done when: holding the key drains the node server-side, releasing stops it, and a request from across the map does nothing.

S3 — The break

[NICK — by hand] Depletion to zero breaks the node, and the break pays the week's first physical dividend. A chunk pops out with a small launch impulse, settles, sleeps, and finally goes net-dormant, which means it drops off the network once it has come to rest. That whole arc is the bounded Tier-2 life — the manual's safe physics budget — that every loose chunk in this game lives, start to finish. Respawn the node with FTimerManager (the engine's built-in timer scheduler). What exactly is a chunk allowed to do between pop and sleep? → doctrine-physics.html#chunk-contract. Done when: the node breaks, pops one chunk that comes to rest still on both clients, and respawns on its timer.

[FLEET] The fleet builds a second resource variant copied from your node pattern, and the point is to prove that the pattern replicates — in both senses of the word. Alongside it, SFX import scripting and CC0 (public-domain) sound sourcing arrive ahead of S5.

S4 — The carry

[NICK — by hand] Server_Grab attaches the chunk to your character, and because this is plain replicated attachment at Tier 2, there is no rigid-body drama to manage. The carry is what brings the waddle. It needs a replicated CarriedWeight and a GetMaxSpeed() override applied on both sides, so the client predicting your movement and the server correcting it agree about exactly how slow greed makes you. Deposit at a collection crate, count the haul with a debug counter, and take drop-on-command as the freebie it is. Why must both sides compute the same max speed? → doctrine-physics.html#cmc. Done when: carrying visibly slows you on every screen, and a chunk dropped by one client deposits from the hands of another.

S5 — The proof and the juice

[NICK — by hand] Wire the node, the pop, and the deposit into UJuiceComponent v0: floating text, a cha-ching, confetti. Sort the channels as you wire. The confetti and the cha-ching ride moments, fired once and free to miss, while the node's health and the chunk on the ground ride facts. Which channel does a visible thing belong on? → concept-netcode.html#facts-vs-moments. Then run the exam below yourself — that part is the rule, not a suggestion. Switch on net emulation, the editor's simulated lag and packet loss, at 100 ms just to feel what the round-trip costs. That 100 ms stays on permanently from week 3. How do I switch it on? → shelf.html#epic-net-emulation. Friday Clip #002 is the waddle. Done when: the cheater test fails safe, the juice fires on both clients, and the clip is posted.

[FLEET] JuiceKit widget boilerplate and the floating-text pool arrive before the session, because S5 is wiring, not widget plumbing.

⚑ Fun payload — the JuiceKit is born

Player: when the node breaks, a physical chunk pops out with a little launch impulse and a confetti burst. On top of that comes the payoff fifteen years of LitRPG promised you: floating combat text, +3 Ferrocrystal! drifting up in mono type. Then you grab the chunk and do the waddle, because carrying visibly slows you, and a capsule-person waddling a comically large rock to a crate is charm at Tier-2 prices. Numbers going up, loudly, plus physical comedy: a complete game. Builder: the feedback is UJuiceComponent, one reusable component (floating text, sound stinger, particle pop, a hitstop hook for later), and every mechanic you ever build gets plugged in the day it's born. That makes it fun engine #1, and also just good architecture. What does the JuiceKit grow into? → fun-engines.html#juicekit

What this week teaches

Three ideas, all taught by one rock. The deep versions live on the concept pages, while the two silent failures are linked from the exact sessions where they bite.

The model. The node's health lives on the server, and each client holds only a copy that it reacts to through that same OnRep_Health hook. That hook is where the crack VFX, the UI tick, and the JuiceKit pop all hang. Every verb this game will ever have is this same shape again: a request goes up, a fact comes down. What does one harvest look like end to end, wire and all? → concept-netcode.html#authority. The server changes facts; clients react to facts changing.

Facts and moments. The JuiceKit is born riding the second of the wire's two channels, the one called moments. A moment is fire-and-forget narration that a late joiner is allowed to miss, while the node's health and the chunk on the ground are facts, guaranteed to converge on every client that ever looks. The sorting question for everything visible you build from now on is simply this: would a join-in-progress client need to see it? The deep answer is taught at concept-netcode.html#facts-vs-moments.

The exam. The proof of internalization is the cheater test, and you run it on yourself. From a client, call the harvest logic locally, using a debug key wired straight to the C++ that bypasses the RPC entirely. If the world changes, you have just built a single-player game with extra steps. If instead nothing happens, it is because the client never had the authority to change anything, and you now understand Unreal networking better than most people do after forty tutorial-hours. From here it's a habit, not a ceremony. The full procedure, run on every server-authoritative verb you ever add, lives at concept-netcode.html#cheater-test.

⚠ Gotcha seeds
  • Forgot GetLifetimeReplicatedProps → the property compiles, runs, and silently never replicates → the trace.
  • An RPC does nothing — no error, no log → check ownership first, always → concept-netcode.html#ownership.
  • A reliable RPC per held frame = buffer overflow = mystery disconnects → concept-netcode.html#reliability.
  • OnRep_* never auto-fires on the server or listen host (a server a player also plays on) for its own changes — call the handler manually after changing the value → concept-netcode.html#repnotify.
  • Chunks vibrating on clients = you skipped the sleep step. A chunk at rest is furniture, not a simulation → doctrine-physics.html#chunk-contract.
  • Editing a header with Live Coding (hot-reload without a restart) active = weirdness. Header changes mean a full restart, so accept that early.
