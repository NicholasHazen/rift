By Friday you will have a UE 5.8 project running two clients against a dedicated server in PIE, a pushed repo, and an agent that builds the code from the command line — and you will know where code lives in an Unreal project because you read it yourself. None of it is the game; all of it is the workshop, which is why this week gets the tightest box in the manual.

▣ Gate — definition of done

The character runs in PIE with 2 clients plus the dedicated-server option — the week's test topology is PIE + dedicated from the very first Play. The repo is pushed and a fresh clone opens clean in the editor. An agent builds from the CLI and correctly answers "where is player input handled in this project?" Clip #001 is posted. Setup week does not get an encore.

An empty shop, and you hanging tools. The only goal is a running project, a healthy repo, and an agent that can build the code — which is exactly why the box is hard. The trap this week: perfecting the workshop instead of using it. You'll know you're in it when editor settings are open before the gate passes. The move: you get oriented by needing things.

The project is OneMoreRock — the exit decision in three words. The codename names the .uproject and the primary C++ module, and it exists before the project does because renaming a UE project later is misery. The register carries the dated decision; it doesn't pre-commit the Bit, which keeps its week-3 date.

What this week teaches

Five ideas, all of them orientation — where an idea has a deeper home, its paragraph carries the one link; the other two are already the whole week-0 lesson.

The two worlds. An Unreal project splits into Source/, where C++ schema lives, and Content/, where binary assets live — the split is why Git LFS exists and why agents can author content without ever touching the schema layer. Your template read-through is the first walk through both. Where exactly does everything live, and why does the split match how you already build software? → concept-anatomy.html#two-worlds

The build toolchain. UnrealBuildTool drives every build; on this Mac the entry point is Build.sh, not the Build.bat most tutorials name; and Live Coding exists — knowing that much is this week's whole requirement. The exact commands land in CLAUDE.md §Build the day the fleet proves them, so they're never folklore.

PIE test topologies. The Number-of-Players field and the dedicated-server option are the week's entire networking lesson: from the very first Play, there is a server, and it isn't you. Which test topology tells the truth about which bugs — and when does the convenient one lie? → concept-netcode.html#topologies

Enhanced Input. The Third Person template is built on Enhanced Input — mapping contexts and input actions rather than hard-wired key bindings — and week 2's ability binding routes through it. This week you only need to know it exists and recognize it in the template code you're reading.

The build host. One question this week seeds instead of answers: this Mac develops the game fine, but it cannot produce the Linux dedicated server or the Windows clients that weeks 6–7 need. Deciding where shipping builds happen is cheap now and expensive in week 7, so OQ-03 carries a week-1 date — and it takes your sign-off, not a fleet default. Why can't this Mac ship week 7's artifacts? → concept-shipping.html#build-host

⚠ Gotcha seeds
  • The Unreal Project Browser creates <Location>/<ProjectName>/ and errors if that folder already exists — create at a temp location, quit, move the contents (.uproject, Source/, Content/, Config/) into the repo root, reopen, and answer Yes to the "rebuild missing modules" prompt — it recompiles from Source/ and opens clean. If it bites differently, it's the first entry in UE-GOTCHAS.md.
  • No full Xcode with its license accepted → C++ project creation fails; Command Line Tools alone are not enough, and the fix is a 10+ GB download you don't want to discover at the create step.
  • Epic publishes no Xcode-requirement row for 5.8; the engine's own Engine/Config/Apple/Apple_SDK.json is the authority. Check it right after install and pin Xcode to what it blesses — an Xcode newer than the engine expects is a known build-breaker, not a free upgrade.
  • First open of a fresh C++ project compiles the module, then thousands of shaders — 15–30 minutes of progress bars is normal, not a hang. Start it, then make the IDE call while it churns.
  • compile_commands.json goes stale after module or header changes — re-run the generate command, or the fleet navigates last week's code.
  • Do not source-build the engine this week — the launcher binary is the point. Source is week 7's problem, on whatever hardware answers OQ-03.
