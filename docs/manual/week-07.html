The sessions

Five sessions: four move the project onto rented hardware, while the fifth fills it with friends and ducks. Each one ends on an artifact you can look at — a screenshot, a name, a command, a clip — because the grind week pays in receipts, one per evening.

S1 — The clone

[NICK] Week 0's GitHub link pays out now. On the build host — the machine OQ-03 chose — clone the engine source and start the engine build. It is hours of machine time, a start-it-and-walk-away job, so kick it off first and draft week 8's bot-client plan while the fans scream. Where can builds physically happen, and which lane carries what? → concept-shipping.html#build-host. Done when: the engine build is running on the build host and the bot-client plan is a note in the repo.

S2 — First cook

[NICK] Add the Server target, run the first cook, and stand the pair up on LAN (your home network): the Linux server from the build host, and packaged clients for the platforms week 6's census named. The week's first receipt arrives tonight — a cooked server booting a console full of your game, no editor anywhere in sight. Done when: a packaged client plays against the cooked server on LAN and the screenshot of that console is in the repo.

S3 — The room

[NICK] Provision the VPS, whose vendor and spec are OQ-12, decided at the moment of purchase. Then deploy the server artifact, write the systemd unit (the Linux service definition), open the firewall, and connect from home. The ceremony lands tonight, not Friday: name the server something worth saying out loud, and set the MOTD, the server's welcome banner, that will greet Friday's friends by name. Done when: you connect from home to a machine with a name, and it greets you with yours.

S4 — One command

[FLEET] Automate the line: one command turns a clean checkout into built, cooked, packaged, deployed artifacts. The fleet owns this pipeline from now on — and where the artifacts live between builds is OQ-13, due before the pipeline runs twice. What invocation should the wrappers encode? → concept-shipping.html#buildcookrun

[NICK] Review the wrapper like you own it, because you do: it cuts a build every week from here to Early Access. Done when: one command produces and ships the artifacts while your hands stay off the keyboard.

S5 — The night

[NICK] Friends connect from packaged clients to the VPS over the real internet and play a full session, Chronicle and all. When it ends, the celebration lands: Ducks 200 on the Remote, the game's tuning console. The server takes its first genuine load with an audience watching. Done when: the gate passes and Clip #008 — ducks — is posted.

⚑ Fun payload — the Duck Test

Builder: the celebration for the first WAN session is a console command on the Remote, Ducks 200, which rains two hundred physics-simulated ducks onto the server. It is your first genuine stress test, and the beginning of a proud tradition of load-testing tools that are also toys. Player: friends connecting to a real server for the first time get greeted by name in the MOTD you set in S3. Small thing; lands big.

Two hundred simulated ducks hit the server at once — which physics tier are they?

Tier 1 — cosmetic. No duck's resting spot is a fact anyone needs, and no player pushes one while it simulates, so the flock carries zero gameplay authority. It is free to sleep, despawn, and disagree between screens. The ducks measure the budget; they never spend it. How do the two sorting questions land there in ten seconds? → doctrine-physics.html#tier-test

What this week teaches

This week teaches almost nothing new, and that is on purpose. This page is the run-book, while the vocabulary lives on concept-shipping.html. What follows is three orientations, one link each.

The engine from source. Epic's launcher binaries omit the dedicated-server target, so shipping a server means building the engine from source — once per engine version, on the build host, hours of machine time you start and walk away from. Why can't the launcher engine ship a server? → concept-shipping.html#why-source

The pipeline, in your vocabulary. Cooking bakes editor-format assets into optimized platform format, and packaging bundles cooked content and binaries into something shippable. That is five stages total, each one a thing you already do for a living under a different name. What are the five stages, translated? → concept-shipping.html#pipeline

The one command. RunUAT BuildCookRun drives the whole line, and S4 wraps it in scripts/ so nobody ever types it again. One command in, artifacts out. What does every flag mean? → concept-shipping.html#buildcookrun

⚠ Gotcha seeds
  • It runs in PIE (Play In Editor) but is missing from the packaged build, because the cooker bakes only what's referenced. → concept-shipping.html#pipeline
  • Your DGX Spark is an ARM machine, but UE desktop and client builds are x86-64 — a different chip architecture. The Spark's Stage-1 jobs are orchestration, bot-fleet control, and (soon) Chronicle-polishing LLMs, not engine builds. Don't burn a session discovering this the hard way. → concept-shipping.html#spark
  • Sprawl gravity: there is always one more thing to make elegant about a build pipeline (concept-shipping.html). Ship ugly. The box is the feature.
