Concept · Shipping & infrastructure

From editor to fleet

When to (re)read this: whenever you like, because this page has the fewest prerequisites in the manual. Read it once before the build-host decision lands in week 1, and again the week before week 7 turns it into commands.

The project spends its first six weeks as an editor toy: it runs where it was written. Shipping is the act of turning it into artifacts that run somewhere else — and every stage of that trip is something you already do for a living, wearing an Unreal name. This page states the one hardware fact the plan cannot ignore, then maps the pipeline into your own vocabulary. From there it annotates the one command that runs all of it, and explains why shipping a server means building the engine from source. Last, it paints the lane markings for the machines involved.

Where builds can physically happen

Start with the fact, because it gates weeks 6 and 7 and it is cheapest to handle now: the dev PC can produce both artifacts week 7 promises. Week 7 ships a Linux dedicated server (the game's headless host process) and Windows clients. Win64, plain 64-bit Windows, is the PC's native platform, so the client half is covered by definition. The server half is covered too, because Epic's Linux cross-compilation toolchain — a kit for building Linux binaries on a Windows host — runs on exactly this kind of machine. Install the engine-matched toolchain, and "package a Linux server plus Windows client" is a command, not a sentence. One catch survives the good news. The Server target, the build recipe for the dedicated server, exists only in a source-built engine (the engine fact below). A source engine, in turn, is a couple hundred gigabytes and hours of compile living beside the engine the Epic launcher already installed.

None of this touches daily work, though. The PC stays the development machine: it runs the editor, it runs PIE (Play In Editor — running the game inside the editor) with the dedicated-server checkbox ticked, and it hosts every session between here and week 7. The Win64 client it packages natively is also the artifact most of the roster runs. Week 6's playtest still doubles as a census of what your friends actually play on, before week 7 promises them artifacts. The census matters here because a macOS client is the one artifact this desk doesn't produce, and if the census asks for one, the house Mac covers it.

So where do shipping builds happen? The pragmatic path is to let the dev PC double as the build host: the source engine goes in beside the launcher engine, and engine builds run overnight, never during a session. The ideal path is containerized CI, or continuous integration, on Epic's official Linux images, where a build becomes a job you submit instead of a machine you own. The Unreal Containers project on the shelf, the manual's resource page, documents that whole practice. My recommendation: the PC doubles for now, containers when the deploy outgrows scp. The pipeline below is identical either way — only the room it runs in changes.

The figure paints the lane markings, one card per machine. The dev PC is where development lives, and — pending OQ-03 — where builds may live too. On it the editor and PIE run nightly, Win64 clients package natively, and the Linux server compiles from the source engine. The standing rule over all of it is simple: engine builds happen overnight, never mid-session. The Mac is the bench, so it packages a macOS client if week 6's census asks for one, and it touches neither the Linux server nor a Win64 client. The Spark only conducts: it drives the bot fleet and, eventually, the Chronicle's local models, but it never runs an engine build, because it is ARM. The VPS (a rented Linux box) sits at the far end of the line, where it runs the server artifact and builds nothing. Four machines, four lanes, and every week-7 surprise lives in a lane violation.

The pipeline, translated

The pipeline is five stages, and you have run all five a thousand times under different names. It opens with the build, which is the ordinary compile step where UnrealBuildTool compiles the Server and Client targets. Next comes the cook, where the cooker bakes every asset from .uasset editor format into the target platform's optimized format — this is the one stage without a common name in ordinary CI, so call it asset compilation.

The cook works by reference, and that is the detail that eventually bites. Starting from the maps the project names, it walks the reference graph and bakes only what it can reach, which means an asset loaded by a bare string path is invisible to that walk and quietly absent from the artifact.

After the cook, the stage assembles binaries and cooked content into a runnable directory layout. Then the package packs the thousands of loose cooked files into .pak archives and bundles the result into something shippable. Last comes the deploy, which is scp (secure copy — moves files over SSH) and a systemd unit (Linux's service manager that keeps processes running) on a rented Linux box. Put your own names to the five and the mystery evaporates: a build step, an asset-compile step, an assembly, a zip, and a release.

The figure draws the line. Five stations run left to right: build, cook, stage, package, deploy. Written underneath each is your own name for it — compile, asset compile, assemble, zip, release. Source code and content enter at the left, and the finished artifacts, a Linux server and a Win64 client, exist from the package station onward. Deploy is what carries the server artifact onto the VPS and puts it under systemd. One command drives the whole line, and it is the next section's subject.

Now run three real changes through the line, because the line is really three routes. The first route is a C++ change. You tighten the range check in Server_Harvest's validation, which lands at the build stage, so UnrealBuildTool recompiles the targets and everything downstream reruns — the full line, both artifacts. The second route is an asset change. The honk stinger gets a longer doppler tail, and because nothing compiled, the cooker simply re-bakes one .uasset, so the change rides the line from cook onward. The third route is a number, and it is the one that never enters the pipeline at all. One command, annotated

Every stage answers to one conductor: RunUAT, the Unreal Automation Tool, which lives in the engine's Engine/Build/BatchFiles/ directory, and its BuildCookRun command. This is the canonical invocation, the one the scripts/ wrappers encode:

RunUAT.bat BuildCookRun -project=OneMoreRock.uproject -platform=Win64 -clientconfig=Development -server -serverplatform=Linux -serverconfig=Development -build -cook -stage -pak -archive -archivedirectory=Builds/

Read it flag by flag. -project names the game. -platform=Win64 sets the client artifact's operating system, and -clientconfig=Development its build configuration — Development until there is a reason to say Shipping. -server -serverplatform=Linux -serverconfig=Development says: also produce the dedicated server, targeting Linux, same configuration.

Then come the verbs, in pipeline order. -build compiles the targets with UnrealBuildTool. -cook bakes the content for each named platform. -stage assembles binaries and cooked content into a runnable layout under Saved/StagedBuilds. -pak packs the loose cooked files into .pak archives. -archive -archivedirectory=Builds/ copies the finished artifacts out to a clean folder — the thing that actually ships. One caveat closes the loop: the invocation only succeeds on a host that can build what it names, which is why the build host is this page's first section.

The command is long, which is why nobody types it. It gets wrapped in a script in scripts/ that takes nothing and produces artifacts, owned and maintained by the fleet and run weekly from week 7 to Early Access. The contract is the whole standard: one command in, artifacts out. Ugly scripts are fine; un-automated steps are not.

Why the launcher engine can't ship a server

An engine fact, stated once so it never surprises you: the engine installed by the Epic launcher cannot build dedicated servers. The launcher ships precompiled engine binaries, and those binaries omit the Server target — your project can declare a Server.Target.cs, but compiling it means recompiling engine modules, and that requires the engine's source. So shipping a server means linking your Epic account to GitHub, cloning the engine, and building it: hours of machine time, a start-it-and-walk-away job, not a watch-it job. It runs once per engine version, on the build host — and it is most of why the build host must exist at all. The shelf's engine-source entry is the exact click path when week 7 arrives.

The Spark's lane

The DGX Spark looks like the obvious build host, because it is the beefiest machine in the house. But it is disqualified by one thing its power can't fix: its architecture is ARM. Unreal's desktop clients and Linux dedicated servers are x86-64 territory, and no amount of patience cross-compiles around that usefully in Stage 1. So the Spark's jobs are the ones that fit the silicon: orchestration, conducting the week-8 bot fleet, and eventually the Chronicle-polishing LLMs (what does the Chronicle want a local model for? → fun-engines.html#chronicle). What it must never touch is an engine build, so don't burn a session discovering that limit the hard way. The Spark conducts; it does not carry.
