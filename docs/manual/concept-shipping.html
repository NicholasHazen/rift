Concept · Shipping & infrastructure

From editor to fleet

When to (re)read this: whenever — this page has the fewest prerequisites in the manual. Read it once before the build-host decision lands in week 1, and again the week before week 7 turns it into commands.

The project spends its first six weeks as an editor toy: it runs where it was written. Shipping is the act of turning it into artifacts that run somewhere else — and every stage of that trip is something you already do for a living, wearing an Unreal name. This page states the one hardware fact the plan cannot ignore, maps the pipeline into your own vocabulary, annotates the one command that runs all of it, explains why shipping a server means building the engine from source, and paints the lane markings for the machines involved.

Where builds can physically happen

Start with the fact, because it gates weeks 6 and 7 and it is cheapest to handle now: this Mac cannot produce either artifact week 7 promises. Week 7 ships a Linux dedicated server and Windows clients. Epic's Linux cross-compilation toolchain runs on Windows hosts, not macOS; packaging a Win64 client requires a Windows machine outright. "Package Linux server plus Windows client," issued from this desk, is a sentence, not a command.

None of this touches daily work. The Mac stays the development machine: the editor, PIE with the dedicated-server checkbox, every session between here and week 7 — and it can package a macOS client natively, which is one reason week 6's playtest doubles as a census of what your friends actually play on, before week 7 promises them artifacts. The gap opens only at shipping time, and only for the two platforms that matter most.

So where do shipping builds happen? The pragmatic path: a home-lab x86 Linux box, which builds the engine from source and the Linux server natively — and the server artifact is the one the whole plan leans on — plus a Windows machine or containerized builds for the Win64 clients. The ideal path: containerized CI on Epic's official Linux images, where a build is a job you submit instead of a machine you own; the Unreal Containers project on the shelf documents that whole practice. My recommendation: the Linux box now, containers when the deploy outgrows scp. The pipeline below is identical either way — only the room it runs in changes.

This is a register question with a date, and the one entry there that takes your sign-off rather than a fleet default. What's on the ballot, and when is it due? → register.html#oq-03. Week 6 checks the friends' platforms; week 7 executes whatever you chose. Decided in week 1 it costs a conversation; discovered in week 7 it costs the week.

The pipeline, translated

The pipeline is five stages, and you have run all five a thousand times under different names. Build: UnrealBuildTool compiles the Server and Client targets — the ordinary compile step. Cook: the cooker bakes every asset from .uasset editor format into the target platform's optimized format — asset compilation, the one stage without a common name in ordinary CI. It cooks by reference: starting from the maps the project names, it walks the reference graph and bakes what it can reach — an asset loaded only by a bare string path is invisible to that walk and quietly absent from the artifact. Stage: staging assembles binaries and cooked content into a runnable directory layout. Package: packaging packs the thousands of loose cooked files into .pak archives and bundles the result into something shippable. Deploy: scp and a systemd unit on a rented Linux box. A build step, an asset-compile step, an assembly, a zip, a release.

The figure draws the line. Five stations run left to right — build, cook, stage, package, deploy — with your names for them written underneath: compile, asset compile, assemble, zip, release. Source code and content enter at the left. The finished artifacts, a Linux server and a Win64 client, exist from the package station onward; deploy carries the server artifact onto the VPS and puts it under systemd. One command drives the whole line, and it is the next section's subject.

One command, annotated

Every stage answers to one conductor: RunUAT, the Unreal Automation Tool, which lives in the engine's Engine/Build/BatchFiles/ directory, and its BuildCookRun command. This is the canonical invocation, the one the scripts/ wrappers encode:

RunUAT.sh BuildCookRun -project=OneMoreRock.uproject -platform=Win64 -clientconfig=Development -server -serverplatform=Linux -serverconfig=Development -build -cook -stage -pak -archive -archivedirectory=Builds/

Read it flag by flag. -project names the game. -platform=Win64 sets the client artifact's operating system, and -clientconfig=Development its build configuration — Development until there is a reason to say Shipping. -server -serverplatform=Linux -serverconfig=Development says: also produce the dedicated server, targeting Linux, same configuration.

Then the verbs, in pipeline order. -build compiles the targets with UnrealBuildTool. -cook bakes the content for each named platform. -stage assembles binaries and cooked content into a runnable layout under Saved/StagedBuilds. -pak packs the loose cooked files into .pak archives. -archive -archivedirectory=Builds/ copies the finished artifacts out to a clean folder — the thing that actually ships. One caveat closes the loop: the invocation only succeeds on a host that can build what it names, which is why the build host is this page's first section.

The command is long, which is why nobody types it. It gets wrapped: a script in scripts/ that takes nothing and produces artifacts, owned and maintained by the fleet, run weekly from week 7 to Early Access. The contract is the whole standard: one command in, artifacts out. Ugly scripts are fine; un-automated steps are not.

Why the launcher engine can't ship a server

An engine fact, stated once so it never surprises you: the engine installed by the Epic launcher cannot build dedicated servers. The launcher ships precompiled engine binaries, and those binaries omit the Server target — your project can declare a Server.Target.cs, but compiling it means recompiling engine modules, and that requires the engine's source. So shipping a server means linking your Epic account to GitHub, cloning the engine, and building it: hours of machine time, a start-it-and-walk-away job, not a watch-it job. It runs once per engine version, on the build host — and it is most of why the build host must exist at all. The shelf's engine-source entry is the exact click path when week 7 arrives.

The Spark's lane

The DGX Spark looks like the obvious build host — it is the beefiest machine in the house — and it is disqualified by its architecture: ARM. Unreal's desktop clients and Linux dedicated servers are x86-64 territory, and no amount of patience cross-compiles around that usefully in Stage 1. The Spark's jobs are the ones that fit the silicon: orchestration, conducting the week-8 bot fleet, and — eventually — the Chronicle-polishing LLMs (what does the Chronicle want a local model for? → fun-engines.html#chronicle). Never engine builds. Don't burn a session discovering this the hard way. The Spark conducts; it does not carry.
