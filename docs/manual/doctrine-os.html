The operating system

This page holds two things: eight standing rules and the machinery of a working week. The rules are guardrails, each one placed where a specific cliff is. The machinery — the session ritual, the Friday close, the bad-week floor — is how two-hour evenings compound into a shipped slice instead of leaking away at the edges. The other doctrines decide what gets built; this one decides whether the building survives contact with a calendar.

The eight rules

Each rule below gets a spoken paragraph — the rule, its practice, and why it protects you — and then one more pass, compressed, on the card at the bottom. The numbers are load-bearing. Every page in this manual cites them bare ("rule 5 says the checkbox stays on"), so a rule keeps its number for life. If a ninth rule ever earns its place, it takes the number nine; nothing renumbers.

Rule 1 · Problem-first media

Never open a video or a doc without a written question it must answer. Video gets one hour a week, batched into Friday's learning hour. The why is blunt: video feels like progress and produces almost none. An hour of watching leaves nothing in the repo unless a question was waiting to be answered. The shelf is this rule applied to the manual itself: every resource on it is filed under the question it answers, so the question comes first by construction. What already has a sanctioned answer? → shelf.html

Rule 2 · The session ritual

A session is a 10/100/10 block: ten minutes loading context, one hundred minutes on one problem, and ten minutes committing and writing the tomorrow-note. That note is one line in docs/tomorrow-note.md naming the next session's first concrete action. A session without a note isn't closed; that is the whole mandate, and it is non-negotiable. The full machinery — the open, the shape, the one-sentence close — is this page's next section.

Rule 3 · The 90-minute valve

When a problem has ground for ninety minutes, stop pushing and pull the valve. The exits are pre-chosen, because deciding on a fallback while stuck is itself a task, and that task won't happen. So this page states the law, while the procedure lives on a page built to be read on one screen at minute 91. One wall must never eat a session; momentum is the resource. What are the three exits, in order? → stuck.html

Rule 4 · First-of-pattern by hand

Five firsts are yours to write by hand: the first replicated property, a variable the server syncs to every client; the first RPC, a function call fired across the network; the first GameplayAbility, an action like a skill or an attack; the first StateTree, an AI state machine; and the first DataAsset, a typed content object on disk. Agents replicate each pattern only after that first example exists, because they generate UE code that compiles and is subtly wrong about authority — the server-or-client question of who owns each fact. Your mental model is the only firewall, which is why every replicated property and every RPC in the codebase ships with a one-line authority explanation you can personally vouch for. How do you prove the explanations are true rather than hopeful? → concept-netcode.html#cheater-test

Rule 5 · Multiplayer from hour one

PIE, which runs the game right inside the editor, launches at least two clients plus the dedicated-server option from the very first Play. That dedicated server is a headless host with no player of its own. Network emulation — 100 ms latency and 2% packet loss — stays permanently on from week 3. That is a checkbox, not folklore. The editor's Play settings carry a network-emulation section with per-profile latency and loss values, so you set the profile once and stop thinking about it. Single-player-only code is a bug written today and found in week 8, at scale. Which test topology lies about which bugs? → concept-netcode.html#topologies

Rule 6 · C++ systems, Blueprint wiring

State, logic, and replication all live in C++. Blueprints — the engine's visual scripting — handle only the wiring, meaning asset hookup, tuned values, and UI layout. Game content like tuning, spawn tables, and word lists flows from JSON into DataAssets and DataTables — DataTables being spreadsheet-shaped rows of typed data. The reason is fleet-shaped: C++ and JSON are text, so your agents can read, diff, and author them, while a Blueprint is an opaque binary no diff can see into. Where exactly do the two worlds split? → concept-anatomy.html#two-worlds. And how does tuning stay text end to end? → concept-anatomy.html#tuning-as-text

Rule 7 · Friday = playtest & learn

On Friday the build gets played — bots or humans — then measured, then clipped, and the week closes with Monday's tomorrow-note already written. The learning hour lives here too: rule 1's video budget spends itself on Friday, against the week's accumulated questions. All of this operationalizes a belief you hold: games are meant to be played. The full close, both tiers of it, is printed below at the Friday close.

Rule 8 · Gray-box till week 10

Until week 10 you build with capsules, cubes, and free sound effects — no art tasks, no trips to Fab, Epic's asset marketplace, and no "just looking." The Fun Mandate exists so this rule costs nothing to obey: charm comes from committed bits, juice, and text, none of which need a mesh. Art-before-fun is procrastination wearing a render pipeline. Where does graybox charm actually come from? → doctrine-fun.html

The session ritual

The open is three clicks. The first lands on the mission board, where the save-file window says where you stand and names the next concrete action — the first line of docs/tomorrow-note.md, rendered live so the front door never rots. The second is the W-number in the masthead strip, which every page renders as a link to the current week's page. The third is the first session heading without a done-mark, and that heading carries tonight's work and its done-when. The target is under five minutes from opening the manual to the first keystroke, because deciding what an evening is for is the most expensive thing you can do at 9pm.

The block itself is the 10/100/10 shape. The first ten minutes load context — the week page, the note, the last commit. The hundred minutes that follow all go to one problem — not the week, not the roadmap, one problem. The last ten close the session: you commit, then write the tomorrow-note naming the next session's first concrete action. Re-entry is where ADHD sessions die. The tomorrow-note parks the car on a downhill slope.

The figure draws the block the way the perf page draws a server tick — one bar, spent in slices. The left slice is the ten minutes of loading context, the middle is the hundred minutes on one problem, and the right is the ten minutes of closing. Two marks outrank the slices. One is the dashed line at minute ninety-one of the problem, where the valve opens. The other is the moss line at the far edge — the tomorrow-note, the close's definition of done. A session without the note isn't closed.

The close costs one sentence: say "close the session." The fleet drafts the tomorrow-note from the session's commits, updates the save file, appends the session row — and the clip row, if one exists — to clips.html, and commits. Then you edit one line and approve. The mandate stays; the cost drops to near zero.

The Friday close

Friday is rule 7 run as a procedure, and the procedure comes in exactly two sizes — both printed here, so no Friday ever has to improvise its own standard. The full close is the good-week shape, while the floor is the law that keeps a bad week from breaking the chain.

The full close

  1. Playtest. Run the build with bots or humans on the topology named by the week's gate — the server-and-client arrangement it requires.
  2. Measure. Record what the gate counts. Done is an observation, not a feeling.
  3. Clip. Record twenty seconds of this week's build and post it.
  4. Approve the paperwork. The fleet drafts the changelog row — one honest sentence naming what shipped, what slipped, and what surprised — then runs the gate audit against the week page's definition of done and sharpens the next week's page. You read, correct, and approve.
  5. Tomorrow-note. Write Monday's first concrete action in one line, before the laptop closes.

A passed gate flips its glyph on the mission board's tracker and earns a gate-cleared window in the archive; the honest sentence lands in changelog.html. Only the next week's page sharpens — the pages beyond it stay sketches until the evidence that sharpens them exists. And any fun payload the week didn't ship rolls forward to the next one — payloads never silently die. Where does fun debt get tracked? → fun-engines.html#fun-debt

The bad-week floor

This is printed law, so a depleted Friday never has to negotiate with itself. The minimum viable close is twenty seconds of anything on screen — a bug is a valid clip — plus a one-line tomorrow-note. The fleet back-fills the rest on Monday: the changelog row, the session bookkeeping, the tidy-up. The floor is not a lowered standard. It is the recognition that the chain matters more than any single link, and one minute of Friday keeps the clip archive, the note, and the save file unbroken. A bad week that hits the floor still counts.

Scheduling reality — This plan assumes ~10–12 focused hours across five-ish 2-hour blocks. You have a day job shipping EMURJ and a family, so some weeks will give you only six hours. The contract: the calendar flexes, the gates don't. A week that takes two weeks is the plan working. A gate that quietly shrinks is the plan dying.
