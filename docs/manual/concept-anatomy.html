Where everything lives

When to (re)read this: in week 0, right after the template project compiles and you're staring at a folder of generated classes — and again any time you're about to add a class, a property, or a tuning value and can't say in one sentence where it belongs.

Unreal hands every game the same skeleton: a fixed cast of framework characters, and a hard wall between code and content. Placement is not taste — it is two questions with procedures behind them. Which character owns this fact? Which side of the wall does this thing live on? This page teaches both procedures, then follows the wall to its consequence: a tuning pipeline where game balance is text your fleet can author.

The cast

The netcode page seats you at the table: the server is the DM, clients ask, facts flow down. Now meet who is actually sitting there. Every Unreal game — shooter, farm sim, rift-extraction game — gets the same troupe whether it wants one or not: a GameMode, a GameState, and, per player, a PlayerController, a PlayerState, and a Pawn. You never design this cast. You cast your facts into it. And "where should this code live?" is never answered with a folder path; it is answered with "which character owns this fact?"

The procedure behind that answer is two questions long, because netcode already settled the third. Who is allowed to change a fact is standing law — the server, always. Why can't the client just change it? → concept-netcode.html#authority. The two live questions are these: who must be able to read this fact, and what must it survive? Every member of the cast is one specific combination of those two answers. Learn the combinations and placement stops being judgment and becomes lookup.

The GameMode is the referee's rulebook. It exists only on the server — and "never replicates" undersells it: clients do not hold a stale copy, they hold no copy; the object is never spawned on their machines at all. That makes it the most private room in the house — the right home for the rules of the session (spawning, win conditions, what happens when the rift closes) and for data players should never hold. It also makes it a trap: a fact stored on the GameMode is a fact 24 screens can never render, no matter how politely they ask.

The GameState is the scoreboard bolted to the tavern wall. One per world, replicated to everyone, alive for the whole session. Anything all 24 players must read at once belongs here — the rift timer, instability, the public face of the ire meter. If the GameMode is the DM's private notes, the GameState is what the DM writes on the whiteboard.

Each player owns a private pair. The PlayerController is the hands and eyes — input lives here — and it exists in exactly two places: the server and the owning client. Player B never has a copy of player A's controller, which is why nothing stored on it can ever be shown to the rest of the table. It is also the root of your voice, the ownership chain every Server RPC routes up. Who may speak, exactly? → concept-netcode.html#ownership. The PlayerState is the character sheet pinned to the party board: per player, replicated to everyone, and it survives respawn — the survival rule that half the placements below turn on. The pawn dies; the sheet persists.

The Pawn — for anything that walks, its Character subclass — is the body. It replicates to every client close enough to care; the engine quietly stops shipping bodies that are far away, which is one of the few mercies that keeps 24 players affordable. What does distance actually save, in numbers? → concept-perf.html#cost-model. Facts stored on the body die with the body. That is a filter, not a hazard: put on the Character exactly the facts that should vanish at death, and nothing else.

NPCs complete the cast. A goose is a Pawn possessed by an AIController — a brain in a jar, server-only, pluggable into any body. Because the brain never replicates, a goose's decision-making costs the wire nothing; only its body does. What lives inside the jar, and why the horde is built this way? → concept-horde.html#brains.

The figure below draws the cast as an existence grid — six rows for the cast, three columns for machines: the server, client A (the player who owns the private pair), and client B (a bystander). A solid box with a check means the object exists on that machine; a dashed outline means it never does. Read the rows: GameMode — one solid box, server only. GameState — solid all the way across. A goose's AIController — server only, same row-shape as the GameMode. Two of the six rows never leave the server, and that column is where the game keeps its secrets.

The ire meter must be visible to all 24 players: which cast member holds it? Write it down.

Run the two questions. Who must read it: everyone — which eliminates the private rooms at once; the GameMode cannot even be asked, and a PlayerController would show it to one player and nobody else. What must it survive: the whole session — which rules out every body. The GameState holds it. One subtlety is worth keeping: the pool itself — the minting, the spending, the director's budget — lives in a server-side subsystem, and the value the players see is a mirror written onto the GameState, precisely because the machinery's natural home never leaves the server. What shape does that machinery take? → concept-ire.html#shapes.

Now drill the rest of the slice through the same two questions, one sentence each. The session timer: read by everyone, alive as long as the session — GameState. Kill credit — say, for dropping Gerald — must outlive both Gerald and your own next death, so it goes on the sheet: PlayerState. The haul a player has banked is the same shape — read by the whole table, survives respawn — PlayerState again. Carry-slow, the waddle of a player hauling too many chunks, is a fact about one specific body and should die with it: the Character and its movement component. And the spawn tables the director buys waves from are data no client may ever hold — server-only, the GameMode's side of the table. What do the spawn tables actually buy? → concept-ire.html#spawn-tables.

That is the whole procedure, stated precisely: name the fact, ask who must read it and what it must survive, and the owner falls out — private and session-long is GameMode territory, public and session-long is GameState, public and per-player is PlayerState, private and per-player is PlayerController, public and mortal is the Pawn. Don't pick a home for the code; pick an owner for the fact.

Two worlds

The cast answers where a fact lives at runtime. The second placement question is about the repo, and it splits the project into two worlds with different physics. Source/ is C++ — text: diffable, reviewable, fleet-legible, yours. Content/ is binary — meshes, maps, Blueprints, .uasset everything — and binary files do not diff and do not merge; they can only be locked, copied, and replaced. One repo, two different version-control physics living under one roof.

The relationship between the worlds is the thing to internalize, because it runs backwards from a library. The engine is a framework you subclass, not a library you call: C++ defines classes — AResourceNode, the tuning struct, the cast members above — and the editor makes tuned asset instances of them on the Content side: your code as the schema, assets as its rows.

If that sentence sounds familiar, it should: this is the EMURJ metafield architecture all over again. Schema in code, content as structured data, and the people who author content never touch the schema layer. You have run a business on this pattern — you know where it bends, where validation has to live, and what happens when someone edits the schema casually. That experience is not decoration; it is the fleet's operating theory for the whole project. Agents author spawn tables, tuning files, and word lists — structured data validated against a schema you wrote — and never hold a binary asset in their hands.

Two standing rules are downstream of this wall, and their rationale lives here. Rule 6 — C++ for systems, Blueprints for wiring — is not aesthetic: C++ and JSON are text, so the fleet can read, diff, author, and review them; Blueprints are opaque binaries, so reviewing a Blueprint diff is two screenshots and a prayer. Logic that drifts into Blueprints exits the fleet's field of vision, and in a one-person studio that means it exits review entirely. Git LFS is a week-0 requirement for the binary world's sake: .uasset and .umap files bloat plain git history and can never be merged anyway, so LFS versions them as blobs behind lightweight pointers and the repo stays clonable in week 12. Keep the game's soul on the text side of the wall.

Tuning as text

The wall has a consequence the whole plan leans on: if the soul stays on the text side, then game balance — ire per harvest tick, wave budgets, decay curves, spawn weights — is a set of files agents can author and you can review. The promise underneath is specific: edit the text, see the change, and it has to stay true not just in the editor but on a packaged Linux server, mid-playtest, from week 7 onward. There are two honest ways to get text into a running game, and they pay differently — the pragmatic path, the ideal path, then the recommendation.

The pragmatic path is runtime-loaded JSON. Define a USTRUCT for the tuning block — schema in code, again — and let FJsonObjectConverter deserialize a plain JSON file into it at startup, with the file living under Saved/ or Config/, outside the cooked content entirely. Add one Remote command — ire.ReloadTuning — that re-reads the file into the live struct, and "edit JSON, see change" stays true everywhere: on the packaged VPS the tuning file is just a file on the server's disk, so the loop is edit, reload, watch the waves change — zero cook, zero deploy, zero recompile. The cost is that JSON can only carry what text can carry: numbers, curves, names. A string that claims to name an NPC class is a promise nothing checks until the wave tries to spawn.

The ideal path is editor-imported DataTables. Author the same content as JSON or CSV, import it in the editor, and it becomes an asset with real, checked references — a spawn-table row points at an actual NPC class, and a broken reference fails loudly at import instead of quietly mid-siege. The price is the wall itself: an imported table is a .uasset now, binary-world, and every change rides import, cook, deploy before a packaged server sees it. Free during editor weeks. A real bill on the VPS.

My recommendation: split by what the data references. Scalars and curves — ire rates, decay, wave budgets, storm timing — ride runtime JSON and stay hot-reloadable on the packaged server, because those are the knobs playtest nights will want turned mid-session. Anything that references an asset — spawn tables pointing at NPC classes — rides DataTables and accepts the cook, because a mistyped class name discovered mid-wave is worse than a deploy. Write the bill down now so week 7 is a deploy week instead of a discovery week: on the VPS, tuning-file edits are live; spawn-table edits ride the pipeline. How does a change actually reach the VPS? → concept-shipping.html#pipeline. Where does the Remote that carries the reload command live? → fun-engines.html#remote. A knob you have to recompile isn't a knob.

The figure draws the split as two lanes. The top lane is the hot one, drawn in ember because it answers to you mid-session: edit the JSON, run ire.ReloadTuning, and the next wake uses the new numbers — in PIE and on the packaged VPS alike, because the file is just a file on the server's disk. The bottom lane rides the wall: the same kind of text, but importing it turns it into a .uasset and walks it into the binary world, so every change pays cook and deploy before a packaged server sees it. Scalars and curves ride the top lane; anything that names an asset rides the bottom, and buys checked references with the fare.

⚠ Symptom index
  • The value is right on the server and missing on every client → the fact lives in a server-only room; nothing there can reach a screen → the cast.
  • Kill credit or haul resets when a player respawns → it was stored on the body; the sheet survives, the body doesn't → the cast.
  • A per-player fact renders for its owner and nobody else → it sits on the PlayerController, which only two machines have → the cast.
  • The tuning change works in PIE and the packaged server ignores it → check which pipeline the value rides: runtime JSON ships as a file, DataTables ship in the cook → tuning as text.
  • An .uasset merge conflict → there is no merge; binary files are lock-and-copy, and someone's work is about to be redone → two worlds.
