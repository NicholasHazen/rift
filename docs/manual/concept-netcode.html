Concept · Netcode & replication

The server is the dungeon master

When to (re)read this: once, start to finish, before week 1's first session — before you hand-write your first replicated property — and again any time the world looks right on one screen and wrong on another.

Unreal multiplayer is one idea wearing eight outfits: the server runs the real game, and everyone else is asking politely. This page builds the whole idea from the dungeon table up — who decides, who may speak, how the mail travels, what must survive a late joiner, and how to prove you got it right. It is the longest listen in the manual. Hear it through once; raid it by anchor forever after.

The table

Picture the table. The server is the dungeon master. It keeps the one true copy of the world — every node's health, every goose's position, every chunk on the ground, every point of ire in the pool. The clients are players sitting around that table. A player never reaches across and edits the DM's notes. A player says what they would like to do — "I harvest the vein" — and the DM checks the rules, changes the world, and narrates back what is now true.

In engine terms: a client sends a Server RPC — a remote procedure call, the polite request. The server validates the request and mutates true state. Changed facts flow back down to every client through replication. And when a fact changes on a client's copy, the client reacts to the change through a hook called RepNotify — "node health dropped, play the crack effect." Your client is never running the game. It is rendering its most recent copy of the DM's notes, and asking nicely when it wants the notes to change.

The figure below is the shape to memorize. One box on top: the server, which owns true state, validates, and narrates. Two boxes underneath: clients, which predict, react, and render. One thin arrow runs up — a request, an RPC, drawn in storm purple, because a request is a question the world has not answered yet. Arrows run down — facts, replication, drawn in ember, because facts are the world's answers landing. Requests go up. Facts come down. Nothing else crosses the wire.

One harvest, traced end to end

Here is week 1's entire curriculum in a single button press. You stand at a vein and hold the harvest key. Your client subtracts nothing — the node's health is not yours to touch. Instead, a Server RPC goes up: Server_Harvest, "I'd like to harvest this." Exactly where that RPC is allowed to live is the next section's whole subject, and it is not a detail.

The server receives the request and checks it like the client is lying — is the node alive, is this player in range — because someday a client will be. The checks pass, and the server ticks the node's Health down. Health is a replicated property: declared UPROPERTY(ReplicatedUsing=OnRep_Health) and registered in GetLifetimeReplicatedProps. Forget that registration and you meet the classic first bug of every UE networking career: the property compiles, runs, and silently never replicates.

The engine notices the changed fact and ships the new value down to every client whose copy is stale. On each client, OnRep_Health fires, and the client reacts: crack VFX, a UI tick, the JuiceKit pop, +3 Ferrocrystal! drifting up in mono type. That includes your own screen. Your reward arrives one round-trip after your button press, because your celebration was never yours to authorize. The division of labor in one sentence — and the sentence is 80% of multiplayer game programming: the server changes facts; clients react to facts changing.

One casting note. The troupe that holds these facts — GameMode, GameState, PlayerState, PlayerController, Pawn — has a page of its own, and "where should this code live?" is always answered by "which character owns this fact?" Where does everything live? → concept-anatomy.html#cast.

⚠ The Salient warning

One warning aimed specifically at you: this is not the deterministic-lockstep world you built for Salient. Nothing here is deterministic; there is no shared simulation to keep in sync. There is one authoritative state and many eventually-consistent views of it. Different religion. Don't syncretize.

Who may speak

The table has a rule the diagram doesn't show, and it is the number-one way week 1 dies in silence: not every actor is allowed to send requests. When a client connects, the server builds it a private chain of actors. The connection owns a PlayerController — the hands and eyes. The PlayerController possesses a Pawn — the body. And anything whose Owner is set into that chain joins it: the chunk your character grabs becomes, for networking purposes, part of your voice.

That chain is the whole of your voice. A Server RPC routes to the server only when it is called on an actor the calling client owns. Called on anything else, it does not error, does not warn, does not log in any way you'll notice at 9pm. It is dropped. Each player speaks only through their own character sheet — shouting requests at the furniture is ignored.

The figure draws the chain. On the left, the owned spine runs top to bottom: the connection owns the PlayerController, the PlayerController possesses the Character, the Character owns the chunk it carries. An ember arrow runs from the Character to the server: an RPC called anywhere on that spine routes. On the right sits AResourceNode — owner: none, just a rock in the world. A dashed line runs from the node toward the server and dies on the way: an RPC called on it from a client is dropped without a sound.

Server_Harvest has two candidate homes — AResourceNode or your Character — and one of them silently never runs when a client calls it: which, and why?

The node version is the trap. A client calling Server_Harvest on the node is shouting at the furniture: no client owns a rock, so the call routes nowhere and reports nothing. The RPC lives on an actor you own — your Character or your PlayerController — and takes the node as a parameter: Server_Harvest(AResourceNode* Target). The server then validates the argument the way it validates everything else — is the target real, alive, in range — because a parameter is just another thing a client can lie about. And this section's law, worth writing on the inside of your eyelids before week 1: if an RPC does nothing, check ownership before anything else.

How the mail travels

Requests and facts are mail, and the engine offers two postage classes. A Reliable RPC is guaranteed and ordered: the engine buffers it, re-sends until the far side acknowledges, and delivers it in order relative to the other reliable calls on the same channel. That guarantee has a price — the buffer is finite. A client that queues reliable RPCs faster than the wire can drain them overflows the buffer, and the engine's response to overflow is to disconnect the client. An Unreliable RPC is fired once and forgotten; if the packet dies, nothing re-sends it.

There is a quiet third class, and you have already met it: replicated properties. A property is not mailed per change — the engine keeps shipping the latest value until every client converges on it. You get a guarantee of convergence, never a play-by-play. Hold that thought; it becomes the fine print of the RepNotify section.

The design rule that falls out of the postage prices: RPCs should be shaped like events, not streams. Week 1's harvest key is the canonical case. The wrong build sends a reliable Server_Harvest every frame the key is held — thirty-plus guaranteed, ordered, buffered messages a second, per player, forever. A reliable RPC per held frame is a disconnect machine under rule 5's own lag emulation: at 100 ms and 2% loss, the buffer backs up and the engine starts ejecting your friends. The right build sends two events — Server_StartHarvest on press, Server_StopHarvest on release — and the server ticks the drain in between, because the drain is a change to facts and facts belong to the DM. RPCs carry events, not streams.

The sorting rule for postage, then: requests ride reliable Server RPCs — they are rare and they matter. Cosmetic one-offs ride unreliable mail, which is the next section's subject. And anything that must stay true rides a replicated property, because convergence is exactly the guarantee state wants.

The campaign log and the table talk

A DM produces two kinds of narration, and they are not interchangeable. Some things go in the campaign log: the wall has 40 health left, the party holds six chunks, the extractor stands. Some things are table talk: the goose's honk dopplers past, confetti bursts, the room laughs, and the moment is gone. The log must reach everyone — including the player who joins at the next session. The table talk only matters if you were there, and a missed line must cost nothing.

The engine has one mechanism for each. Facts ride replicated properties with RepNotify — state, guaranteed to converge, visible to anyone who ever looks. Moments ride unreliable multicast RPCs — the server tells every connected client "play this now," once, fire-and-forget. (GAS arrives in week 2 with GameplayCues, which are this same idea with better plumbing.) The discriminator is a single question, and you should ask it about every visible thing you ever build: would a join-in-progress client need to see it? Yes: it is a fact — put it in a replicated property. No: it is a moment — multicast it and never think about it again.

Run week 1 through the question. The node's health: a fact — a late joiner must see a half-mined vein at half health. The confetti burst when the node breaks: a moment — a joiner who missed the party missed the party. The chunk lying on the ground afterward: a fact again — the rock exists, someone can carry it, its rest position must be true everywhere. The chunk's full journey from pop to sleep is physics doctrine, not netcode. What contract keeps loose chunks from becoming a simulation? → doctrine-physics.html#chunk-contract.

This distinction is the JuiceKit's load-bearing wall. Every pop, stinger, and floating number the JuiceKit throws is a moment — triggered by facts changing, but never itself state. Lose a pop to packet loss and nobody's haul is wrong; somebody's world was momentarily quieter. Juice that accidentally becomes state — a celebration a client must see for the game to proceed — is a bug wearing a party hat. What exactly does the JuiceKit ship, and when? → fun-engines.html#juicekit.

One more rule lives here, and the Chronicle depends on it. Discrete events with narrative weight — first blood, the biggest yeet, who carried what through the portal — travel as a replicated event list or RPCs, never reconstructed from OnRep sequences. The reason is in the next section's first clause: replication is allowed to skip the middle of any story. If the debrief must tell the story, log the story as it happens. How the log becomes prose at debrief → fun-engines.html#chronicle.

A closing note on scale, because facts have a bill: the server does not actually ship every fact to every client every tick — actors out of range go irrelevant, sleeping actors go dormant, and both save the server real money at 24 players. What do relevancy and dormancy buy, in numbers? → concept-perf.html#cost-model.

The fine print on facts

RepNotify looks like an event system: value changes, function fires. It is not an event system. It is a convergence system with a callback, and the difference is four clauses of fine print. Each clause has a failure smell; learn the smells now and you will diagnose in minutes what otherwise costs evenings.

Clause one: OnReps coalesce. The engine ships latest values, not journeys. If health goes 3 → 0 → 3 between two network updates — a break and an instant respawn, say, or any A→B→A — the client receives 3, compares it to the 3 it already has, and fires nothing. The break never happened, as far as that client can prove. The smell: an effect that usually plays but sometimes doesn't, and gets flakier as load and lag rise. The law: replication syncs state; it does not deliver events. Events that matter ride the event list or an RPC — which is exactly why the Chronicle's rule in the previous section exists.

Clause two: ordering is not guaranteed across properties or actors. Two facts changed together on the server may arrive in either order on a client. Live, you rarely notice — changes trickle one at a time and arrive as they happened. But an OnRep that reads a second property while reacting to the first is gambling on arrival order, and the gamble mostly pays until a late joiner receives both facts in one bunch, shuffled. The smell has a name you will use for years: works live, wrong after JIP — join-in-progress.

Clause three: initial replication fires against defaults. When an actor first replicates to a client, its OnReps fire comparing the incoming values to a freshly constructed actor's defaults — not to any history. A wall at damage stage three arrives to a late joiner as one notify: default to three. If that OnRep narrates the transition — plays the flinch, spawns the debris — your joiner watches an hour-old injury happen in front of them. Write every OnRep to converge, not narrate: "make the visuals match this value now," never "play what just happened."

Clause four: REPNOTIFY_Always. By default, a client skips the notify when the incoming value equals the one it already holds. Between test swings, Gerald's health is set back to full; a client that already reads full hears nothing, and the reset never renders. Registering the property with REPNOTIFY_Always (in GetLifetimeReplicatedProps) forces the notify on every received write, equal or not. Reach for it when re-asserting a value is itself meaningful — respawns, wave resets, anything where "again" is information.

And the clause that outranks the other four, because it bites the machine you test on every night: OnReps never auto-fire on the server, or on a listen host for its own changes. Replication is something that happens to clients; the machine that changed the value received nothing, so nothing fires. If the server needs the reaction — and for anything gameplay-relevant it usually does — call the handler manually right after changing the value. This is also the first member of a bug family that gets its own section below: code that behaves differently depending on which machine is the server.

The player who joins at session two

The law first, because it is the headline of this whole page applied to time: state must live in replicated properties, not in the memory of having-been-there.

A player who joins the campaign at session two does not get a dramatic reading of session one. The DM hands over the current sheets, and everything true must be readable off the sheets. A join-in-progress client is exactly that player: it receives the world through initial replication of replicated state, and through nothing else. It never heard your multicasts. It never ran your OnReps in the order the story happened — clause two and clause three above are its daily weather. Its BeginPlay runs into a world that is already old, which is why JIP failures hide in "I set that on the client in BeginPlay" logic. If the camp only looks right to players who watched it get built, replication is subtly broken — and at 24 players with rolling arrivals, this bug class is fatal.

So the manual makes it an exam, run deliberately in week 5 while the world is still small and the fixing is cheap. The audit:

  1. Build a camp with two clients — walls at mixed damage stages, an extractor mid-hum, chunks loose on the ground.
  2. Connect a third client, cold.
  3. Walk the camp with the third client, slowly, looking at everything.
  4. Diff what it sees against server state: wall count, damage stages, extractor state, every loose chunk, the ire stage, the scoreboard.
  5. Every mismatch is a fact that lives somewhere other than a replicated property. Fix it, and write the fix into UE-GOTCHAS.md — each one is institutional memory now.

The proof

Everything above is theory until you try to cheat your own game. The cheater test is the proof of internalization, and it is a procedure, not a vibe — run it on every server-authoritative verb you ever add, the week you add it:

  1. Pick the newest verb: harvest, place a wall, deal damage.
  2. From a client, call the mutation logic directly — a debug key wired straight to the C++, bypassing the Server RPC entirely.
  3. Watch the server's world, and a second client's screen.
  4. Pass: nothing happens. The node doesn't tick, the wall doesn't spawn, the second client sees stillness. Fail: any change visible beyond your own screen — a client of yours has authority it should not have.

If the world changes, you built a single-player game with extra steps. If nothing happens, the model held: the client shouted, and the DM's notes never moved. The test pairs with a standing habit: every replicated property and every RPC in this codebase ships with a one-line authority explanation, and the cheater test is how you audit that the explanations are true rather than hopeful.

Three tables

The same campaign can be run at three different tables, and every table tells you different truths. Knowing which table you tested at is not optional bookkeeping — it is part of every gate's definition of done, because a claim of "works" means nothing until it says where.

PIE with the dedicated-server checkbox is the everyday rig and rule 5's standing order: two or more clients, plus a headless server the editor spawns for you. It gives you a true client/server split at zero setup cost, which makes it the honest default for every session. Its lie: everything runs on one machine, so latency is zero until you turn on net emulation — which is why 100 ms and 2% loss are permanently on from week 3. Where is that standing order written as law? → doctrine-os.html#rule-5.

A listen server makes one player's game the server — the DM playing a character at their own table. It is how the week 6 Tailscale nights run, because friends need something to connect to and a VPS doesn't exist yet. The listen host is a special creature, and its semantics are worth learning once, as a family: the host has authority and a local player. On the host's own pawn, HasAuthority() and IsLocallyControlled() are both true at once. OnReps never fire for the host's own changes — the fine-print clause from above, now wearing a player's face and wondering where its health bar went. The host's input never crosses a wire, so the host feels zero latency and proves nothing about lag. Code paths that work because the host is both server and player are bugs — the topology just hasn't told you yet.

A dedicated server — the VPS, from week 7 on — is the shipped shape and the only table that doesn't lie. No machine is both DM and player; every client eats real latency; the Linux build surfaces everything platform-shaped. The bug family it exists to catch has a name in this manual: passes-on-listen, breaks-on-the-VPS — every assumption that quietly leaned on the host being a player, invoiced all at once, in week 7, unless you hunted them earlier. How a build actually gets onto that VPS → concept-shipping.html#pipeline.

⚠ Symptom index
  • My property compiles, runs, and never replicates → you skipped GetLifetimeReplicatedProps — see the trace, then the Epic property-replication doc on the shelf.
  • My Server RPC does nothing, no error, no log → ownership, before anything else.
  • Clients start disconnecting when someone holds a button → a streaming reliable RPC; reshape it into events → reliability.
  • The effect usually plays but sometimes doesn't, worse under lag → OnReps coalesce → the fine print.
  • Works live, wrong after a late join → ordering and initial replication → the fine print and the exam.
  • A reset or respawn never shows on clients → REPNOTIFY_Always → the fine print.
  • The server (or the host) never reacts to its own change → OnReps don't auto-fire where the change is made → the fine print.
  • Passes on the listen server, breaks on the VPS → the host was both server and player → three tables.
